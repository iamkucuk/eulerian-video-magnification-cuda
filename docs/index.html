Abstract

Eulerian Video Magnification (EVM) reveals subtle temporal changes in video that are invisible to the naked eye. The pulse of blood under skin, the breathing of an infant, the vibration of machinery. Unlike feature-tracking approaches, EVM treats each pixel as a time series, applies a temporal bandpass filter, amplifies the frequencies of interest, and reconstructs the video with those invisible changes made visible.

This project ports the MIT SIGGRAPH 2012 reference implementation from MATLAB/Python to raw CUDA C++, achieving a 269x speedup over the Python/NumPy baseline while producing output validated bit-for-bit within an RMSE of 0.01. The implementation uses no high-level GPU frameworks (no PyTorch, no CuPy) — every kernel is hand-written CUDA C++ with manual memory management, coalesced access patterns, and templated FP16 storage. The full pipeline runs device-resident with zero per-frame host-device transfers, and fits in 12 GB VRAM using the FP16 path.

Method

Every EVM variant follows the same four-stage pipeline. Each stage maps to one or more CUDA kernels, with the entire pipeline running device-resident: 

input video (T frames, H x W, RGB uint8)
   |
   1. COLOR    BGR u8 -> NTSC YIQ float (per-pixel matrix multiply)
   2. SPATIAL  Gaussian downsample (color) OR Laplacian pyramid (motion)
   3. TEMPORAL Bandpass filter along time (FFT / Butterworth / IIR)
   4. AMPLIFY  Multiply by alpha, add back, render to RGB
   |
output video (magnified)

The port validates against the Python baseline (evm/), which is itself validated against MIT's published output videos. Per-kernel tolerances range from 1e-6 (pure arithmetic) to 1e-4 (cuFFT), with end-to-end RMSE held below 0.01.

Performance

Compute-only (pipeline stages, excluding video I/O), measured on NVIDIA A100-80GB and Tesla P100-16GB:

Pipeline Python CPU CUDA FP32 CUDA FP16 Throughput
Color (pulse) 10,350 ms 72 ms (144x) 84 ms (124x) 1.08 Gpx/s
Motion (breathing) 46,255 ms 209 ms (222x) 172 ms (269x) 0.88 Gpx/s

At 1080p (1920×1080), a single A100 can process 20 parallel color streams or 14 parallel motion streams at 30 FPS in real time. FP16 motion halves VRAM from 23 GB to 12 GB, running on 16 GB consumer GPUs like the Tesla P100 and T4. See the optimization writeup for the full per-stage breakdown across both GPU generations.

Architecture

  • Device-resident pipeline — entire clip staged to GPU once; 50+ kernel launches with zero per-frame transfers
  • Batched spatial kernels — grid.z = M collapses ~35,000 launches into ~50
  • Templated FP16 storage — all kernels compile in both FP32 and FP16; compute stays FP32, storage halves
  • cuFFT plan caching — eliminates per-call autotuning overhead
  • 125 tests — every kernel validated against the Python oracle, including MIT reference comparison

No PyTorch, no CuPy, no Numba. Every kernel is hand-written CUDA C++ with pybind11 bindings. See DESIGN.md for the kernel-by-kernel mapping and tolerance contract.
